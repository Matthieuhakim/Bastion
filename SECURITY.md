# Security Policy

## Supported Versions

This project is still pre-1.0. Security fixes are applied to the latest code on `main`.

## Reporting a Vulnerability

Please do not open public GitHub issues for security vulnerabilities.

Instead:

1. Use GitHub's private vulnerability reporting flow for this repository if it is enabled.
2. If private reporting is not available, contact the maintainer privately before disclosing details publicly.

When reporting a vulnerability, please include:

- A clear description of the issue
- The affected component or package
- Reproduction steps or a proof of concept
- The expected impact
- Any suggested remediation if you have one

You can expect an initial acknowledgement within a few business days. Once the issue is confirmed,
the goal is to coordinate a fix and public disclosure responsibly.

## Scope

Security reports are especially helpful for issues involving:

- Authentication and authorization bypass
- Secret handling or credential exposure
- Proxy request validation and SSRF protections
- Policy engine enforcement gaps
- Audit chain integrity or tamper resistance
- Supply chain risks in release or publishing workflows
